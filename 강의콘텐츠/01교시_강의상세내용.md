## 3. DX 이해하기

### 3.1 DX란 무엇인가?

DX는 Digital Transformation, 즉 디지털 전환을 의미한다.

비개발자에게는 다음과 같이 설명할 수 있다.

> DX는 종이, 엑셀, 전화, 수작업으로 처리하던 업무를 디지털 시스템으로 바꾸는 것이다.

즉, DX는 단순히 컴퓨터를 사용하는 것이 아니다.
업무 방식 자체를 디지털 기반으로 다시 설계하는 것이다.

### 3.2 DX의 쉬운 예시

#### 예시 1. 수강신청 업무

기존 방식:

```text
전화 문의
→ 수강생 정보 수기 기록
→ 엑셀에 옮겨 적기
→ 문자로 안내
→ 입금 여부 수동 확인
```

DX 방식:

```text
온라인 신청서 작성
→ 수강생 정보 자동 저장
→ 자동 문자 발송
→ 입금 상태 관리
→ 관리자 화면에서 확인
```

#### 예시 2. 컨설팅 보고서 작성 업무

기존 방식:

```text
현장 방문
→ 메모 작성
→ 사진 정리
→ 엑셀 분석
→ 한글/워드 보고서 작성
→ PDF 저장
→ 이메일 발송
```

DX 방식:

```text
현장 데이터 입력
→ 시스템 저장
→ 자동 분석
→ 보고서 양식 자동 생성
→ 검토 후 PDF 출력
```

### 3.3 DX의 핵심

DX의 핵심은 “디지털 도구 사용”이 아니라 “업무흐름의 재설계”이다.

| 구분 | 단순 디지털화 | DX |
| --- | --- | --- |
| 의미 | 기존 업무를 컴퓨터로 처리 | 업무방식을 디지털 중심으로 재설계 |
| 예시 | 종이 신청서를 PDF로 받기 | 온라인 신청부터 관리까지 시스템화 |
| 초점 | 도구 사용 | 업무 프로세스 개선 |
| 결과 | 일부 편리함 | 시간 절감, 오류 감소, 데이터 축적 |

## 4. AX 이해하기

### 4.1 AX란 무엇인가?

AX는 AI Transformation, 즉 AI 전환을 의미한다.

비개발자에게는 다음과 같이 설명할 수 있다.

> AX는 디지털화된 업무에 AI를 결합해 생성, 분석, 요약, 추천, 검수, 예측까지 가능하게 만드는 것이다.

DX가 “업무를 시스템으로 바꾸는 것”이라면,
AX는 “시스템이 AI의 도움을 받아 더 똑똑하게 일하게 만드는 것”이다.

### 4.2 DX와 AX의 차이

| 구분 | DX | AX |
| --- | --- | --- |
| 핵심 | 업무의 디지털화 | 업무의 지능화 |
| 주요 도구 | 웹, 앱, DB, 업무시스템 | 생성형 AI, 분석 AI, 자동화 에이전트 |
| 예시 | 수강신청 시스템 | AI가 수강생 문의 유형을 분류 |
| 사람 역할 | 시스템에 입력하고 관리 | AI 결과를 검토하고 판단 |
| 결과 | 업무 효율화 | 의사결정과 실행의 고도화 |

### 4.3 AX의 쉬운 예시

#### 예시 1. 마케팅 콘텐츠 생성

DX 수준:

```text
관리자가 웹사이트에 게시글을 직접 입력한다.
```

AX 수준:

```text
과정 정보를 입력하면 AI가 홍보문구를 생성한다.
관리자가 검토 후 승인하면 Facebook에 자동 게시된다.
```

#### 예시 2. 고객 문의 응대

DX 수준:

```text
문의 내용을 시스템에 저장한다.
```

AX 수준:

```text
AI가 문의 내용을 분석해
“수강료 문의”, “개강일 문의”, “취업 문의”로 자동 분류하고
답변 초안을 생성한다.
```

#### 예시 3. 경영진단 보고서

DX 수준:

```text
매출 데이터를 시스템에 저장한다.
```

AX 수준:

```text
AI가 매출 변화를 분석하고
위험 신호, 개선 과제, 다음 실행전략을 제안한다.
```

## 5. 자동화와 AI 자동화의 차이

### 5.1 자동화란?

자동화는 사람이 반복적으로 하던 일을 시스템이 대신 처리하는 것이다.

예를 들어 다음과 같은 업무는 자동화에 적합하다.

- 매일 같은 양식의 보고서 만들기
- 예약 시간에 게시물 올리기
- 신청자에게 안내 문자 보내기
- 파일명을 규칙에 맞게 바꾸기
- 엑셀 데이터를 정리하기
### 5.2 AI 자동화란?

AI 자동화는 단순 반복작업을 넘어서, AI가 내용을 이해하거나 생성하거나 판단을 보조하는 자동화이다.

예를 들면 다음과 같다.

| 일반 자동화 | AI 자동화 |
| --- | --- |
| 정해진 시간에 게시물 등록 | AI가 게시문구를 생성한 뒤 예약 게시 |
| 신청자에게 문자 발송 | 신청자 유형별로 다른 안내문 생성 |
| 엑셀 합계 계산 | 매출 감소 원인 분석과 개선안 제안 |
| 파일 업로드 | 문서 내용을 요약하고 보고서 초안 생성 |

### 5.3 교육생에게 강조할 점

모든 업무를 AI로 처리할 필요는 없다.

업무는 다음 세 가지로 나누어야 한다.

| 업무유형 | 처리 방식 |
| --- | --- |
| 단순 반복업무 | 시스템 자동화 |
| 문서·콘텐츠 생성업무 | 생성형 AI 활용 |
| 최종 판단·승인업무 | 사람이 결정 |

예를 들어, Facebook 게시물을 자동으로 올리는 것은 시스템이 할 수 있다.
하지만 그 게시물이 학원 브랜드에 적합한지, 과장광고 위험이 없는지, 실제로 게시해도 되는지는 사람이 최종 승인해야 한다.

## 6. 바이브코딩 이해하기

### 6.1 바이브코딩이란?

비개발자에게 바이브코딩은 이렇게 설명하면 된다.

> 바이브코딩은 내가 만들고 싶은 기능과 화면을 자연어로 설명하고, AI 개발도구가 그 설명을 바탕으로 코드를 만들어 주는 개발 방식이다.

즉, 사람이 모든 코드를 직접 쓰는 것이 아니라,
사람이 원하는 서비스를 설명하고 AI가 코드를 제안하는 방식이다.

### 6.2 바이브코딩의 쉬운 비유

#### 비유 1. 건축 설계

집을 짓는다고 생각해 보자.

“예쁜 집 지어줘”라고 말하면 좋은 집이 나오기 어렵다.

대신 다음처럼 말해야 한다.

- 방은 몇 개인가?
- 화장실은 몇 개인가?
- 주방은 어떤 구조인가?
- 창고가 필요한가?
- 주차장은 필요한가?
- 예산은 얼마인가?
- 몇 층 건물인가?
바이브코딩도 마찬가지이다.

“마케팅 AX/DX 서비스 만들어줘”라고 말하면 부족하다.

다음처럼 말해야 한다.

- 누가 사용하는가?
- 어떤 정보를 입력하는가?
- 어떤 결과가 나와야 하는가?
- 누가 승인하는가?
- 실패하면 어떻게 처리하는가?
- 어떤 데이터를 저장하는가?
- 어떤 화면이 필요한가?
#### 비유 2. 신입직원에게 업무 지시하기

AI 개발도구는 매우 똑똑한 신입직원과 비슷하다.

신입직원에게 “홍보 좀 해주세요”라고 하면 결과가 들쭉날쭉하다.

하지만 다음처럼 지시하면 결과가 좋아진다.

```text
이번 주 금요일 개강하는 패션디자인 입문반을 홍보해야 합니다.
대상은 고등학생과 대학생입니다.
Facebook에 올릴 게시글이 필요합니다.
너무 과장된 표현은 피하고, 상담 문의를 유도해 주세요.
게시 전 원장 승인을 받아야 합니다.
```

바이브코딩도 이와 같다.
AI에게 구체적으로 설명할수록 더 좋은 결과가 나온다.

## 7. 바이브코딩에 대한 흔한 오해

### 7.1 오해 1. “이제 개발자는 필요 없다”

그렇지 않다.

바이브코딩은 개발을 쉽게 도와주지만, 복잡한 서비스에서는 여전히 다음 역량이 필요하다.

- 구조 설계
- 보안 설계
- 데이터베이스 설계
- API 연동
- 오류 처리
- 성능 관리
- 배포와 운영
비개발자는 개발자가 되기보다, AI와 개발자에게 정확히 지시할 수 있는 기획자가 되는 것이 중요하다.

### 7.2 오해 2. “한 문장으로 서비스가 완성된다”

“수강생 모집 AX/DX 서비스 만들어줘”라고 하면 겉보기 화면은 나올 수 있다.

하지만 실제 서비스에 필요한 다음 기능이 빠질 수 있다.

- 로그인
- 권한관리
- 학원별 데이터 분리
- 승인 워크플로우
- 게시 실패 재시도
- 이미지 저작권 관리
- API 오류 로그
- 개인정보 보호
- 결제와 구독
- 관리자 기능
따라서 한 문장 지시가 아니라, 기능명세서와 UI 설계서가 필요하다.

### 7.3 오해 3. “AI가 만든 것은 바로 써도 된다”

AI가 만든 코드나 문구는 반드시 검수해야 한다.

AI가 만든 홍보문구는 다음을 확인해야 한다.

- 과장광고는 없는가?
- 사실과 다른 내용은 없는가?
- 수강생에게 오해를 줄 표현은 없는가?
- 개인정보가 포함되어 있지 않은가?
- 저작권 문제가 없는가?
AI가 만든 코드는 다음을 확인해야 한다.

- 실제로 작동하는가?
- 보안 문제가 없는가?
- 데이터가 제대로 저장되는가?
- 다른 사용자의 데이터가 보이지 않는가?
- 오류가 발생했을 때 처리되는가?
## 8. 바이브코딩에서 비개발자의 역할

### 8.1 비개발자는 무엇을 해야 하는가?

비개발자의 역할은 코드를 쓰는 것이 아니다.
비개발자의 역할은 “무엇을 만들어야 하는지 정확히 설명하는 것”이다.

| 단계 | 비개발자의 역할 |
| --- | --- |
| 문제정의 | 어떤 업무를 해결할 것인지 정의 |
| 프로세스 설계 | 현재 업무와 개선 업무흐름 작성 |
| 기능정의 | 어떤 기능이 필요한지 목록화 |
| 화면정의 | 어떤 화면에서 어떤 작업을 할지 정의 |
| 데이터정의 | 어떤 정보를 저장해야 하는지 정의 |
| AI역할정의 | AI가 생성·분석·요약할 업무 정의 |
| 예외처리 | 실패했을 때 어떻게 할지 정의 |
| 검수 | 결과물이 요구사항과 맞는지 확인 |

### 8.2 비개발자가 작성해야 할 핵심 문서

바이브코딩을 잘하기 위해 비개발자가 작성해야 할 문서는 다음과 같다.

| 문서 | 목적 |
| --- | --- |
| 서비스 정의서 | 무엇을 만드는 서비스인지 정의 |
| 요구사항 정의서 | 서비스가 반드시 해야 할 일을 정리 |
| 기능명세서 | 기능별 입력·처리·출력·예외 정의 |
| UI 설계서 | 화면 구성과 사용자 흐름 정의 |
| 데이터 항목표 | 저장해야 할 정보 정의 |
| AI 기능 설계서 | AI가 해야 할 일과 금지할 일 정의 |
| API 연동 흐름도 | 외부 서비스와 연결되는 방식 정의 |
| 보안 체크리스트 | 개인정보·권한·로그·저작권 점검 |
| 개발지시서 | AI 코딩도구에 줄 구체적 지시문 |

## 9. 예시로 이해하는 AX/DX 서비스

### 9.1 예시 서비스: 패션학원 Facebook 홍보 자동화

#### 기존 업무

```text
원장이 과정 정보를 정리한다.
마케팅 직원이 홍보문구를 작성한다.
이미지를 찾거나 만든다.
원장에게 카카오톡으로 확인받는다.
Facebook에 직접 접속해 게시한다.
게시 후 반응을 수동으로 확인한다.
```

#### 문제점

- 매번 문구 작성 시간이 오래 걸린다.
- 승인 이력이 남지 않는다.
- 게시 실패를 바로 알기 어렵다.
- 이미지 출처 관리가 안 된다.
- 어떤 게시물이 효과가 좋았는지 확인하기 어렵다.
#### AX/DX 서비스로 바꾸면

```text
과정 정보 등록
→ AI가 홍보문구 생성
→ 이미지 선택
→ 원장 승인
→ Facebook 자동 게시
→ 게시 실패 시 재시도
→ 성과지표 확인
→ 다음 콘텐츠 개선
```

### 9.2 이 예시에서 DX와 AX 구분

| 기능 | DX 요소 | AX 요소 |
| --- | --- | --- |
| 과정 정보 등록 | 데이터 저장 | 없음 |
| 콘텐츠 관리 | 게시물 상태 관리 | 없음 |
| AI 문구 생성 | 없음 | 생성형 AI |
| 승인 워크플로우 | 업무 시스템화 | 위험문구 AI 검토 가능 |
| Facebook 게시 | API 자동화 | 성과 기반 개선 제안 가능 |
| 성과분석 | 데이터 수집 | AI 리포트 생성 가능 |

## 10. 비개발자가 반드시 알아야 할 10개 개념

### 10.1 사용자

서비스를 사용하는 사람이다.
예: 원장, 마케팅 직원, 승인자, 관리자

### 10.2 권한

누가 어떤 기능을 사용할 수 있는지 정하는 것이다.
예: 직원은 승인 요청만 가능하고, 원장은 최종 승인 가능

### 10.3 데이터

서비스가 저장해야 하는 정보이다.
예: 학원명, 과정명, 게시문구, 이미지, 승인상태

### 10.4 화면

사용자가 보는 작업 공간이다.
예: 대시보드, 콘텐츠 생성 화면, 승인관리 화면

### 10.5 기능

버튼을 눌렀을 때 시스템이 수행하는 일이다.
예: AI 문구 생성, 승인 요청, 예약 게시

### 10.6 상태값

업무가 현재 어느 단계에 있는지 표시하는 값이다.
예: 작성중, 승인대기, 승인완료, 예약됨, 게시실패

### 10.7 API

다른 서비스에 일을 부탁하는 연결창구이다.
예: Facebook API에 “이 게시물을 올려줘”라고 요청

### 10.8 로그

누가 언제 무엇을 했는지 남기는 기록이다.
예: 누가 승인했는지, 언제 게시했는지, 왜 실패했는지

### 10.9 예외처리

문제가 생겼을 때 어떻게 할지 정하는 것이다.
예: 게시 실패 시 3번까지 재시도

### 10.10 MVP

처음부터 완벽한 서비스를 만들지 않고, 가장 중요한 기능만 담은 첫 버전이다.
예: AI 문구 생성, 승인, Facebook 게시까지만 먼저 구현
