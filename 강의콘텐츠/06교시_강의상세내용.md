# 4. 기능이란 무엇인가?

## 4.1 비개발자를 위한 쉬운 정의

기능은 사용자가 어떤 버튼을 누르거나 어떤 행동을 했을 때, 시스템이 수행해야 하는 일이다.

예를 들어 다음은 기능에 해당한다.

- 로그인하기
- 클래스 등록하기
- AI 홍보문구 생성하기
- 이미지 업로드하기
- 승인 요청하기
- Facebook에 예약 게시하기
- 게시 실패 시 재시도하기
- 게시물 성과 확인하기

즉, 기능은 단순한 아이디어가 아니라 사용자의 행동과 시스템의 처리 결과가 연결된 단위이다.

## 4.2 기능이 아닌 것과 기능인 것

| 표현 | 기능 여부 | 설명 |
|---|---|---|
| 마케팅 자동화 | 기능 아님 | 너무 넓은 개념 |
| AI 문구 생성 | 기능 | 사용자가 실행할 수 있는 구체적 기능 |
| 편리한 화면 | 기능 아님 | 추상적인 요구 |
| Facebook 예약 게시 | 기능 | 특정 결과가 있는 시스템 동작 |
| 고객관리 | 기능 아님 | 여러 기능의 묶음 |
| 수강생 목록 조회 | 기능 | 조회라는 구체적 행동 |
| 보안 강화 | 기능 아님 | 정책 또는 비기능 요구사항 |
| 사용자별 권한 설정 | 기능 | 시스템에서 구현해야 하는 기능 |

## 4.3 기능을 잘못 정의하면 생기는 문제

나쁜 기능 정의의 예는 다음과 같다.

> AI 마케팅 기능

이 정의는 다음과 같은 문제가 있다.

- 어떤 마케팅인지 알 수 없다.
- AI가 무엇을 해야 하는지 모호하다.
- 입력값과 출력값이 없다.
- 개발자가 구현 단위를 알 수 없다.

좋은 기능 정의의 예는 다음과 같다.

> 클래스 정보를 기반으로 Facebook 수강생 모집용 홍보문구 3안을 생성한다.

이 정의가 좋은 이유는 다음과 같다.

- 입력값이 있다: 클래스 정보
- 처리방식이 있다: AI 생성
- 출력값이 있다: 홍보문구 3안
- 사용 목적이 있다: Facebook 수강생 모집
- 개발 기능으로 전환 가능하다.

# 5. 사용자 시나리오에서 기능 도출하기

## 5.1 기능은 어디서 나오는가?

기능은 갑자기 떠올리는 것이 아니라, 5교시에서 작성한 사용자 시나리오에서 나온다.

예를 들어 다음 시나리오를 보자.

1. 마케팅 직원이 로그인한다.
2. 대시보드에서 “새 콘텐츠 만들기”를 클릭한다.
3. 홍보할 클래스를 선택한다.
4. 홍보 목적과 톤앤매너를 선택한다.

8. 이미지 보관함에서 이미지를 선택한다.
9. Facebook 미리보기를 확인한다.
10. “승인 요청” 버튼을 클릭한다.

이 시나리오에서 다음과 같은 기능을 도출할 수 있다.

# 7. 우선순위란 무엇인가?

## 7.1 비개발자를 위한 쉬운 정의

우선순위는 어떤 기능을 먼저 만들고, 어떤 기능을 나중에 만들 것인가를 정하는 기준이다.

비개발자에게는 다음처럼 설명할 수 있다.

> 우선순위는 기능의 중요도를 정하는 것이 아니라, 개발 순서를 정하는 것이다.

어떤 기능이 중요하더라도 처음부터 만들지 않을 수 있다. 반대로 단순해 보이는 기능이라도 서비스 작동에 반드시 필요하면 먼저 만들어야 한다.

## 7.2 P0, P1, P2의 의미

| 우선순위 | 의미 | 판단 기준 |
|---|---|---|
| P0 | MVP 필수 기능 | 이 기능이 없으면 서비스가 작동하지 않음 |
| P1 | 정식 서비스에 필요한 기능 | 유료화, 운영 안정성, 사용자 만족에 필요 |
| P2 | 향후 확장 기능 | 있으면 좋지만 초기에는 없어도 됨 |

## 7.3 P0 기능의 기준

P0 기능은 첫 버전에 반드시 들어가야 한다. 다음 질문에 “예”라고 답하면 P0일 가능성이 높다.

1. 이 기능이 없으면 핵심 시나리오가 완료되지 않는가?
2. 이 기능이 없으면 사용자가 서비스를 쓸 수 없는가?
3. 이 기능이 없으면 데이터가 저장되지 않는가?
4. 이 기능이 없으면 보안 문제가 생기는가?
5. 이 기능이 없으면 외부 게시나 자동화가 불가능한가?

### P0 예시

| 기능 | P0인 이유 |
|---|---|
| 로그인 | 사용자 구분과 보안의 출발점 |
| 학원 정보 등록 | 서비스 기본 데이터 |
| 클래스 등록 | AI 문구 생성의 입력값 |
| AI 홍보문구 생성 | 서비스 핵심 가치 |
| 콘텐츠 편집 | AI 결과물을 사람이 수정해야 함 |
| 승인 요청 | 승인 후 게시 구조의 핵심 |
| Facebook 게시 | 서비스 핵심 결과 |
| 게시 실패 저장 | 운영 안정성에 필수 |

## 7.4 P1 기능의 기준

P1 기능은 없어도 MVP는 작동하지만, 정식 서비스나 유료화에는 필요하다.

| 기능 | P1인 이유 |
|---|---|
| 캠페인 관리 | 콘텐츠를 묶어 관리하면 편리하지만 초기 필수는 아님 |

| 프롬프트 템플릿 관리 | AI 품질 개선에 도움 |

# 8. 기능 우선순위 판단 기준

## 8.1 기능 우선순위 6대 기준

| 기준 | 질문 |
|---|---|
| 고객가치 | 사용자가 실제로 필요로 하는가? |
| 핵심흐름 | 핵심 사용자 시나리오에 필요한가? |
| 빈도 | 자주 사용하는 기능인가? |
| 리스크 | 없으면 문제가 발생하는가? |
| 난이도 | 1인 개발자가 구현 가능한가? |
| 수익성 | 유료 기능으로 전환 가능성이 있는가? |

## 8.2 우선순위 점수표

각 기능을 1~5점으로 평가한다.

| 기능명 | 고객가치 | 핵심흐름 | 사용빈도 | 리스크 | 구현용이성 | 수익성 | 총점 |
|---|---:|---:|---:|---:|---:|---:|---:|
| AI 문구 생성 | 5 | 5 | 5 | 3 | 4 | 5 | 27 |
| Facebook 게시 | 5 | 5 | 4 | 4 | 3 | 5 | 26 |
| 이미지 업로드 | 4 | 5 | 4 | 3 | 4 | 4 | 24 |
| 결제 기능 | 3 | 1 | 2 | 2 | 3 | 5 | 16 |
| Instagram 연동 | 4 | 2 | 3 | 3 | 2 | 4 | 18 |

### 해석 기준

- 총점이 높고 핵심흐름에 필요한 기능은 P0 후보이다.
- 수익성은 높지만 핵심흐름에 당장 필요하지 않은 기능은 P1 또는 P2로 둘 수 있다.
- 구현 난이도가 높은 기능은 초기 MVP에서 제외하거나 단순화해야 한다.

# 9. MVP 기능을 정하는 방법

## 9.1 MVP란 무엇인가?

MVP는 Minimum Viable Product의 약자이다.

처음부터 완벽한 제품을 만드는 것이 아니라, 고객이 실제로 핵심 가치를 경험할 수 있는 최소 기능 제품이다.

비개발자에게는 다음처럼 설명한다.

> MVP는 작지만 실제로 쓸 수 있는 첫 번째 버전이다.

## 9.2 MVP의 기준

MVP에 들어갈 기능은 다음 조건을 만족해야 한다.

1. 핵심 고객 문제를 해결한다.
2. 핵심 사용자 시나리오를 끝까지 수행하게 한다.
3. 개발 기간이 지나치게 길지 않다.
4. 사용자가 실제로 테스트할 수 있다.
5. 이후 확장 가능한 구조를 가진다.

## 9.3 MVP에 넣으면 안 되는 기능

초기 MVP에는 다음 기능을 조심해야 한다.

| 기능 유형 | 제외 이유 |
|---|---|
| 채널 과다 확장 | Facebook, Instagram, Naver, Kakao를 한 번에 만들면 복잡함 |
| 고급 대시보드 | 초기에는 기본 지표만 충분 |
| 복잡한 결제 | 고객 검증 전에는 수동 결제도 가능 |
| 디자인 편집기 | Canva 수준 기능은 범위 초과 |
| 자동 댓글 응대 | 고객 클레임 리스크 큼 |
| AI 이미지 생성 고도화 | 저작권·품질 리스크 |
| 다단계 조직관리 | 초기 소규모 고객에게 과도함 |

## 9.4 패션학원 서비스 MVP 예시

### MVP 핵심 흐름

# 13. 기능 우선순위 결정 시 자주 하는 실수

## 13.1 “있으면 좋은 기능”을 P0로 넣는다

### 나쁜 예

> AI 이미지 생성은 멋있으니까 P0입니다.

### 좋은 판단

AI 이미지 생성은 차별화 요소가 될 수 있지만, MVP에서는 이미지 업로드와 선택만으로도 핵심 흐름이 가능하므로 P2로 둔다.

## 13.2 결제 기능을 너무 일찍 넣는다

초기 MVP는 실제 고객이 쓸지 검증하는 것이 우선이다. 결제는 중요하지만, 처음에는 수동 계약이나 계좌이체로도 검증할 수 있다.

## 13.3 채널 확장을 너무 빨리 한다

Facebook, Instagram, Naver Blog, Kakao Channel을 한 번에 연동하면 개발 난이도와 정책 리스크가 급격히 커진다.

초기에는 하나의 채널을 안정적으로 만든 뒤 확장하는 것이 좋다.

## 13.4 관리자 기능을 완전히 빼먹는다

비개발자는 보통 사용자 화면만 생각한다. 하지만 SaaS 서비스에는 운영자 기능이 필요하다.

예시는 다음과 같다.

- 가입 학원 목록 확인
- API 오류 확인
- 사용량 확인
- 게시 실패 모니터링
- 계정 정지
- 플랜 관리

단, MVP에서는 최소 운영 기능만 넣고, 고급 관리자 기능은 P1/P2로 나눈다.

## 13.5 실패 기능을 후순위로 미룬다

“게시 실패 재시도” 같은 기능은 눈에 띄지 않지만, 서비스 신뢰에 매우 중요하다.

특히 자동 게시 서비스에서 실패 대응이 없으면 사용자는 서비스를 믿지 못한다. 따라서 다음 기능은 P0에 가깝다.

- 게시 실패 저장
- 오류 메시지 표시
- 재시도 버튼
- 실패 이력 조회

# 14. 기능 우선순위 판단 질문

교육생에게 각 기능별로 다음 질문을 던지게 한다.

1. 이 기능이 없으면 핵심 시나리오가 중단되는가?
2. 사용자가 매번 쓰는 기능인가?
3. 이 기능이 고객이 돈을 낼 이유와 연결되는가?
4. 이 기능이 없으면 오류나 리스크가 커지는가?
5. 1인 개발자가 MVP에서 구현 가능한가?
6. 외부 API 정책 리스크가 큰가?
7. 나중에 만들어도 되는 기능인가?
8. 이미 다른 무료 도구로 대체 가능한가?
9. 기능을 더 단순하게 만들 수 있는가?
10. 이 기능은 사용자 가치인가, 기획자의 욕심인가?

---

# 핵심 요약

1. 기능은 사용자 행동과 시스템 처리가 연결된 단위이다.
2. 기능은 사용자 시나리오에서 도출해야 한다.
3. 기능명은 짧고 구체적으로 작성해야 한다.
4. P0는 MVP 필수 기능이다.
5. P1은 정식 서비스와 유료화에 필요한 기능이다.
6. P2는 향후 확장 기능이다.
7. MVP는 작지만 실제로 작동해야 한다.
8. 기능 욕심을 줄이는 것이 좋은 기획의 시작이다.
9. 실패 재시도, 권한, 로그 같은 보이지 않는 기능도 중요하다.
10. 기능 목록과 우선순위는 기능명세서 작성의 기초가 된다.
