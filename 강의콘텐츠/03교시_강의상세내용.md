# 3교시 강의안

## AS-IS/TO-BE 업무 프로세스 설계

### 비개발자를 위한 AX/DX 서비스 기획 실습

## 2. 3교시 학습목표

2교시에서는 자동화할 과제를 발굴했다.
3교시에서는 그 과제를 실제 개발 가능한 업무흐름으로 정리한다.

교육생은 3교시 종료 후 다음을 수행할 수 있어야 한다.

- AS-IS와 TO-BE의 의미를 설명할 수 있다.

- 현재 업무흐름을 단계별로 나눌 수 있다.

- 업무 단계별 담당자, 사용도구, 입력값, 출력값을 정리할 수 있다.

- 현재 업무의 병목, 중복, 누락, 오류 지점을 찾을 수 있다.

- AI와 자동화가 들어갈 위치를 표시할 수 있다.

- 사람이 반드시 검토·승인해야 하는 구간을 구분할 수 있다.

- 바이브코딩 개발지시서에 활용할 수 있는 TO-BE 프로세스를 작성할 수 있다.

# 4. AS-IS와 TO-BE의 의미

## 4.1 AS-IS란 무엇인가?

AS-IS는 “현재 업무가 실제로 어떻게 이루어지고 있는가”를 뜻한다.

비개발자에게는 다음과 같이 설명할 수 있다.

> AS-IS는 지금 우리가 일하는 방식이다.
> 누가, 어떤 순서로, 어떤 도구를 사용해, 무엇을 입력하고, 어떤 결과물을 만드는지 정리한 것이다.

### AS-IS 예시

패션학원에서 수강생 모집 게시물을 올리는 현재 방식:

```text
원장이 개강 과정 정보를 정리한다.
↓
직원이 홍보문구를 작성한다.
↓
원장이 카카오톡으로 확인한다.
↓
직원이 이미지를 찾는다.
↓
Facebook에 직접 접속한다.
↓
게시글을 등록한다.
↓
댓글과 좋아요를 수동으로 확인한다.
```

## 4.2 TO-BE란 무엇인가?

TO-BE는 “앞으로 개선된 업무가 어떻게 이루어져야 하는가”를 뜻한다.

비개발자에게는 다음과 같이 설명할 수 있다.

> TO-BE는 자동화와 AI를 적용한 미래의 업무방식이다.
> 어떤 단계는 AI가 돕고, 어떤 단계는 시스템이 자동 처리하며, 어떤 단계는 사람이 승인하도록 다시 설계한 업무흐름이다.

### TO-BE 예시

패션학원 수강생 모집 게시물 자동화 방식:

```text
직원이 클래스 정보를 입력한다.
↓
AI가 홍보문구 3안을 생성한다.
↓
직원이 문구와 이미지를 선택한다.
↓
원장이 시스템에서 승인한다.
↓
시스템이 예약 시간에 Facebook에 자동 게시한다.
↓
실패 시 재시도하고 관리자에게 알린다.
↓
성과지표를 자동 수집한다.
```

## 4.3 AS-IS와 TO-BE의 차이

| 구분 | AS-IS | TO-BE |
| --- | --- | --- |
| 의미 | 현재 업무방식 | 개선된 미래 업무방식 |
| 목적 | 문제 찾기 | 개선안 설계 |
| 중심 질문 | 지금 어떻게 일하고 있는가? | 앞으로 어떻게 일해야 하는가? |
| 초점 | 수작업, 중복, 오류, 지연 | 자동화, AI, 승인, 데이터, 효율 |
| 산출물 | 현재 업무흐름도 | 개선 업무흐름도 |
| 활용 | 문제정의 | 개발지시서·기능명세서 작성 |

# 5. AS-IS 프로세스 작성 방법

## 5.1 AS-IS 작성의 기본 원칙

AS-IS를 작성할 때는 “이상적인 업무”가 아니라 “실제로 하고 있는 업무”를 적어야 한다.

### 잘못된 방식

> “우리는 콘텐츠를 기획하고 검토한 뒤 게시합니다.”

너무 추상적이다.

### 좋은 방식

> “마케팅 직원이 과정명을 엑셀에서 확인하고, 이전 홍보글을 복사해 수정한 뒤, 원장에게 카카오톡으로 보내 확인받고, 다시 Facebook에 직접 접속해 게시합니다.”

실제 업무가 보인다.

## 5.2 AS-IS 작성 항목

| 항목 | 질문 |
| --- | --- |
| 업무 단계 | 어떤 순서로 일이 진행되는가? |
| 담당자 | 각 단계는 누가 하는가? |
| 사용도구 | 엑셀, 카카오톡, 이메일, Facebook, 종이문서 등 |
| 입력값 | 단계별로 필요한 정보는 무엇인가? |
| 출력값 | 단계별 결과물은 무엇인가? |
| 소요시간 | 각 단계에 시간이 얼마나 걸리는가? |
| 문제점 | 어디서 지연, 오류, 누락이 생기는가? |
| 승인 여부 | 누가 확인하고 승인하는가? |
| 기록 여부 | 업무 이력이 남는가? |
| 예외상황 | 실패하거나 반려될 때 어떻게 처리하는가? |

# 6. AS-IS에서 문제 찾기

## 6.1 업무 문제를 찾는 기준

AS-IS를 작성한 뒤에는 문제점을 찾아야 한다.
AX/DX 서비스는 문제를 해결하기 위해 만드는 것이기 때문이다.

| 문제 유형 | 설명 | 예시 |
| --- | --- | --- |
| 중복 | 같은 정보를 여러 번 입력 | 신청자 정보를 엑셀과 문자 시스템에 각각 입력 |
| 지연 | 승인이나 확인이 늦음 | 원장 확인이 늦어 게시가 지연 |
| 누락 | 해야 할 일을 빠뜨림 | 예약 게시를 깜빡함 |
| 오류 | 사람이 실수함 | 날짜, 수강료, 링크 오기재 |
| 비표준 | 사람마다 방식이 다름 | 직원마다 홍보문구 품질 차이 |
| 기록 부재 | 이력이 남지 않음 | 누가 승인했는지 모름 |
| 데이터 단절 | 자료가 흩어짐 | 이미지, 문구, 성과가 각각 따로 있음 |
| 위험 | 법적·보안 문제 | 수강생 사진 동의 없이 게시 |
| 의존성 | 특정 사람만 처리 가능 | 원장만 Facebook 계정 접근 가능 |

## 6.2 문제 표시 방법

AS-IS 업무흐름도 옆에 문제 유형을 표시한다.

예시:

```text
과정 정보 확인
↓
홍보문구 직접 작성 [시간소요 / 품질편차]
↓
원장 카카오톡 검토 [승인 이력 없음 / 지연]
↓
이미지 검색 [저작권 위험]
↓
Facebook 직접 게시 [게시 누락 / 오류]
↓
성과 수동 확인 [데이터 축적 안 됨]
```

# 7. TO-BE 프로세스 설계 방법

## 7.1 TO-BE 설계의 기본 원칙

TO-BE는 현재 업무를 그대로 시스템에 옮기는 것이 아니다.
불필요한 단계를 줄이고, AI와 자동화를 넣고, 사람이 판단해야 할 구간을 명확히 남기는 것이다.

### TO-BE 설계 원칙

- 중복 입력을 줄인다.

- 반복 작업은 시스템이 처리한다.

- 문서·문구 생성은 AI가 초안을 만든다.

- 위험이 있는 업무는 사람이 승인한다.

- 실패 가능성이 있는 업무는 재시도와 알림을 설계한다.

- 중요한 업무는 로그를 남긴다.

- 데이터는 다음 분석에 활용할 수 있게 저장한다.

- 처음부터 모든 기능을 넣지 말고 MVP 범위를 정한다.

## 7.2 TO-BE 작성 항목

| 항목 | 질문 |
| --- | --- |
| 개선 단계 | 앞으로 어떤 순서로 일할 것인가? |
| 자동화 방식 | 시스템 자동화인가, AI 자동화인가? |
| 담당 주체 | 사람, AI, 시스템 중 누가 하는가? |
| 입력값 | 필요한 데이터는 무엇인가? |
| 출력값 | 결과물은 무엇인가? |
| 승인 | 누가 언제 승인하는가? |
| 예외처리 | 실패하면 어떻게 처리하는가? |
| 저장 데이터 | 어떤 이력을 남길 것인가? |
| 알림 | 누구에게 알려야 하는가? |
| 성과 | 어떤 지표를 확인할 것인가? |

## 7.3 TO-BE 정리표

| 단계 | 개선 업무 내용 | 처리 주체 | 자동화 방식 | 입력값 | 출력값 | 예외처리 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 클래스 정보 입력 | 사람 | 시스템 입력 | 과정명, 대상, 기간 | 클래스 데이터 | 필수값 누락 경고 |
| 2 | 홍보문구 생성 | AI | 생성형 AI | 클래스 데이터 | 문구 3안 | API 실패 시 재시도 |
| 3 | 이미지 선택 | 사람 | 시스템 선택 | 이미지 보관함 | 게시 이미지 | 권리정보 누락 경고 |
| 4 | 승인 요청 | 시스템 | 워크플로우 | 콘텐츠 | 승인 요청 | 승인자 미지정 경고 |
| 5 | 승인 처리 | 사람 | 사람 판단 | 미리보기 | 승인/반려 | 수정 요청 |
| 6 | 예약 게시 | 시스템 | API 자동화 | 승인 콘텐츠 | Facebook 게시물 | 실패 시 재시도 |
| 7 | 성과 수집 | 시스템 | API 자동화 | 게시물 ID | 성과지표 | 권한 오류 안내 |

# 8. 사람·AI·시스템 역할 구분하기

## 8.1 왜 역할 구분이 중요한가?

비개발자들이 자동화 설계를 할 때 가장 자주 하는 실수는 모든 일을 AI에게 맡기려는 것이다.

하지만 실제 서비스에서는 다음 세 주체를 구분해야 한다.

| 주체 | 역할 |
| --- | --- |
| 사람 | 입력, 검토, 승인, 최종 판단 |
| AI | 초안 생성, 요약, 분석, 추천, 검수 |
| 시스템 | 저장, 상태관리, 예약 실행, API 호출, 로그 기록 |

## 8.2 역할 구분 예시

| 업무 단계 | 사람 | AI | 시스템 |
| --- | --- | --- | --- |
| 클래스 정보 등록 | 입력 | 없음 | 저장 |
| 홍보문구 작성 | 조건 선택 | 문구 생성 | 결과 저장 |
| 위험문구 확인 | 최종 판단 | 위험표현 검토 | 경고 표시 |
| 콘텐츠 승인 | 승인/반려 | 없음 | 승인 이력 저장 |
| Facebook 게시 | 게시 요청 | 없음 | API 호출 |
| 게시 실패 대응 | 재시도 선택 | 오류 설명 보조 | 실패 로그 저장 |
| 성과분석 | 개선 방향 판단 | 요약·제안 | 지표 수집 |

# 9. 자동화 적용 지점 표시하기

AS-IS와 TO-BE를 비교할 때 다음 표시를 사용하면 좋다.

| 표시 | 의미 |
| --- | --- |
| [AI] | 생성형 AI가 처리 |
| [AUTO] | 시스템 자동화 |
| [HUMAN] | 사람이 처리 |
| [APPROVAL] | 승인 필요 |
| [API] | 외부 서비스 연동 |
| [LOG] | 이력 저장 |
| [RISK] | 리스크 검토 필요 |
| [ALERT] | 알림 필요 |

### 예시

```text
클래스 정보 입력 [HUMAN]
↓
AI 홍보문구 생성 [AI]
↓
위험문구 검토 [AI/RISK]
↓
직원 수정 [HUMAN]
↓
원장 승인 [APPROVAL]
↓
Facebook 예약 게시 [API/AUTO]
↓
게시 실패 시 알림 [ALERT/LOG]
↓
성과지표 저장 [API/LOG]
```

# 11. AS-IS/TO-BE 설계 시 자주 하는 실수

## 11.1 현재 업무를 너무 간단히 적는다

나쁜 예:

```text
홍보글 작성 → 게시
```

좋은 예:

```text
과정 정보 확인
→ 이전 홍보글 검색
→ 문구 수정
→ 이미지 선택
→ 원장 확인
→ 재수정
→ Facebook 접속
→ 게시
→ 성과 확인
```

## 11.2 TO-BE를 너무 거창하게 만든다

나쁜 예:

```text
AI가 모든 마케팅을 자동으로 수행한다.
```

좋은 예:

```text
AI가 홍보문구 초안을 생성하고, 원장이 승인한 콘텐츠만 Facebook에 예약 게시한다.
```

## 11.3 승인과 예외처리를 빼먹는다

나쁜 예:

```text
AI 문구 생성 → 자동 게시
```

좋은 예:

```text
AI 문구 생성 → 직원 수정 → 원장 승인 → 예약 게시 → 실패 시 재시도
```

## 11.4 데이터 저장을 고려하지 않는다

나쁜 예:

```text
게시물을 만든다.
```

좋은 예:

```text
게시물 제목, 본문, 이미지, 승인자, 게시일시, 게시결과, 실패사유를 저장한다.
```

## 11.5 실패 상황을 고려하지 않는다

나쁜 예:

```text
예약 시간에 게시된다.
```

좋은 예:

```text
예약 시간에 게시를 시도하고, 실패하면 오류코드를 저장하며, 최대 3회 재시도하고, 최종 실패 시 관리자에게 알린다.
```

# 15. 3교시 평가 기준

| 평가항목 | 우수 기준 |
