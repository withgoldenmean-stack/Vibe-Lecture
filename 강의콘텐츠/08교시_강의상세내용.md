# 4. 기능명세서란 무엇인가?

## 4.1 쉬운 정의

기능명세서는 “사용자가 어떤 행동을 하면 시스템이 무엇을 처리하고 어떤 결과를 보여줘야 하는지”를 기능별로 정리한 문서이다.

비개발자에게 더 쉽게 말하면 다음과 같다.

> 기능명세서는 개발자와 AI 코딩도구에게 주는 업무 지시서이다.

기능명세서는 단순히 기능 이름만 적는 문서가 아니다. 기능이 실제로 어떻게 작동해야 하는지, 어떤 입력값을 받고, 어떤 처리를 하며, 어떤 결과를 보여줘야 하는지까지 구체적으로 설명하는 문서이다.

예를 들어 “AI 홍보문구 생성”이라는 기능명만 있으면 개발자나 AI 코딩도구는 다음을 알 수 없다.

- 어떤 정보를 입력해야 하는가?
- AI는 어떤 결과를 만들어야 하는가?
- 결과는 몇 개를 생성해야 하는가?
- 결과 형식은 문장인가, JSON인가?
- 금칙어는 어떻게 처리하는가?
- API 실패 시 어떻게 해야 하는가?
- 생성 결과를 저장해야 하는가?
- 누가 이 기능을 사용할 수 있는가?

따라서 기능명세서는 기능명만 나열한 문서를 개발 가능한 수준으로 확장하는 문서이다.

## 4.2 기능 목록과 기능명세서의 차이

| 구분 | 기능 목록 | 기능명세서 |
|---|---|---|
| 목적 | 어떤 기능이 필요한지 나열 | 각 기능이 어떻게 작동할지 설명 |
| 수준 | 간단한 목록 | 개발 가능한 세부 설명 |
| 예시 | AI 홍보문구 생성 | 클래스 정보를 입력하면 AI가 홍보문구 3안을 JSON으로 생성하고 저장 |
| 포함 내용 | 기능명, 우선순위 | 목적, 사용자, 입력값, 처리방식, 출력값, 예외처리, 권한, 로그 |
| 활용 | 기획 범위 파악 | 실제 개발·검수 기준 |

## 4.3 기능명세서가 중요한 이유

| 이유 | 설명 |
|---|---|
| 개발 오해 감소 | 개발자와 AI 코딩도구가 기능을 정확히 이해한다. |
| 누락 방지 | 입력값, 출력값, 예외처리, 권한을 빠뜨리지 않는다. |
| 테스트 기준 | 기능이 제대로 만들어졌는지 검수할 수 있다. |
| 개발 우선순위 관리 | P0, P1, P2에 따라 개발 순서를 정할 수 있다. |
| 바이브코딩 품질 향상 | 자연어 지시가 구체적일수록 AI 코딩 결과가 좋아진다. |
| 유지보수 용이 | 나중에 기능을 수정하거나 확장할 때 기준이 된다. |

# 6. 항목별 작성법

## 6.1 기능 ID 작성

기능 ID는 기능을 관리하기 위한 번호이다.

| 기능 ID | 기능명 |
|---|---|
| F-001 | 로그인 |
| F-002 | 학원 정보 등록 |
| F-003 | 클래스 정보 등록 |
| F-004 | AI 홍보문구 생성 |
| F-005 | 콘텐츠 승인 요청 |
| F-006 | Facebook 예약 게시 |

### 강사용 설명

기능 ID를 붙이면 개발 중 커뮤니케이션이 쉬워진다.

“AI 기능 수정해주세요”보다 “F-004 AI 홍보문구 생성 기능의 출력값을 수정해주세요”라고 말하는 것이 훨씬 정확하다.

## 6.2 기능명 작성

기능명은 짧고 구체적으로 작성한다.

### 나쁜 기능명

- 마케팅 관리
- AI 기능
- 편리한 게시
- 고객관리
- 자동화

### 좋은 기능명

- AI 홍보문구 생성
- 클래스 정보 등록
- 콘텐츠 승인 요청
- Facebook 예약 게시
- 게시 실패 재시도
- 게시 이력 조회

### 작성 원칙

| 원칙 | 설명 |
|---|---|
| 동작이 드러나야 함 | 등록, 생성, 수정, 승인, 게시, 조회 |
| 너무 넓지 않아야 함 | “마케팅 관리”보다 “AI 홍보문구 생성” |
| 사용자가 이해해야 함 | 개발자 용어보다 업무 용어 사용 |
| 한 기능은 한 가지 일을 해야 함 | 생성과 게시를 한 기능으로 묶지 않음 |

## 6.3 기능 목적 작성

기능 목적은 “왜 이 기능이 필요한가”를 설명한다.

### 작성 공식

> 이 기능은 [사용자]가 [업무 목적]을 달성하기 위해 필요하다.

| 기능명 | 기능 목적 |
|---|---|
| 클래스 정보 등록 | AI 홍보문구 생성을 위한 과정 정보를 저장하기 위해 필요 |
| AI 홍보문구 생성 | 마케팅 담당자가 수강생 모집 문구를 빠르게 작성하기 위해 필요 |
| 승인 요청 | 외부 게시 전 원장 검토를 거쳐 과장광고와 오류를 줄이기 위해 필요 |
| 예약 게시 | 지정한 시간에 Facebook에 자동 게시하여 게시 누락을 줄이기 위해 필요 |

## 6.4 사용자 작성

사용자는 이 기능을 실제로 사용하는 사람이다.

| 기능명 | 사용자 |
|---|---|
| 클래스 정보 등록 | 마케팅 직원, 원장 |
| AI 홍보문구 생성 | 마케팅 직원 |
| 승인 처리 | 원장, 승인자 |
| Facebook Page 연동 | 원장, 관리자 |

### 주의

“사용자”라고만 쓰면 안 된다. Owner, Staff, Approver, Viewer처럼 역할을 구분해야 한다.

## 6.5 입력값 작성

입력값은 사용자가 직접 입력하거나, 시스템이 기능 실행을 위해 받아야 하는 정보이다.

### 예시: 클래스 정보 등록

| 입력값 | 설명 | 필수 여부 |
|---|---|---|
| 과정명 | 예: 패션디자인 입문반 | 필수 |
| 모집대상 | 예: 고등학생, 대학생 | 필수 |
| 수업기간 | 예: 8주 | 필수 |
| 개강일 | 예: 2026-07-01 | 선택 |
| 수강료 | 예: 500,000원 | 선택 |
| 과정 강점 | 소수정예, 포트폴리오 제작 등 | 필수 |
| 커리큘럼 | 주차별 수업내용 | 선택 |

### 입력값 작성 시 질문

1. 사용자가 직접 입력하는 정보는 무엇인가?
2. 필수값과 선택값은 무엇인가?
3. 입력 형식은 텍스트, 날짜, 숫자, 파일 중 무엇인가?
4. 잘못 입력하면 어떤 경고를 보여줘야 하는가?
5. 입력값이 다른 기능에서 재사용되는가?

## 6.6 처리방식 작성

처리방식은 사용자가 기능을 실행했을 때 시스템이 내부적으로 해야 하는 일이다.

### 작성 공식

> 사용자가 [행동]을 하면 시스템은 [처리]를 수행한다.

### 예시: AI 홍보문구 생성

1. 사용자가 클래스와 홍보 목적, 톤앤매너를 선택한 뒤 “AI 문구 생성” 버튼을 클릭한다.
2. 시스템은 해당 클래스 정보와 학원 브랜드 톤을 조회한다.
3. 시스템은 OpenAI API에 프롬프트를 전송한다.
4. AI는 홍보문구 3안을 생성한다.
5. 시스템은 결과를 JSON 형식으로 파싱한다.
6. 생성 결과와 사용 모델, 생성 시간을 `ai_generations` 테이블에 저장한다.
7. 사용자에게 생성 결과 3안을 카드 형태로 보여준다.

### 비개발자 설명

처리방식은 “화면 뒤에서 시스템이 하는 일”이다. 사용자가 보는 것은 버튼 하나일 수 있지만, 실제로는 데이터 조회, AI 호출, 결과 저장, 오류 처리 등이 함께 일어난다.

## 6.7 출력값 작성

출력값은 기능 실행 후 사용자에게 보여주거나 시스템에 남겨야 하는 결과이다.

| 기능명 | 출력값 |
|---|---|
| AI 홍보문구 생성 | 제목, 본문, 해시태그, CTA, 위험경고 |
| 승인 요청 | 승인 요청 완료 메시지, 상태값 `review_requested` |
| Facebook 예약 게시 | 예약 완료 메시지, 예약 ID, 예약 시간 |
| 게시 실패 재시도 | 재시도 결과, 성공/실패 상태 |
| 게시 이력 조회 | 게시물명, 상태, 게시시간, 실패 사유 |

## 6.8 예외처리 작성

예외처리는 기능이 정상적으로 작동하지 않을 때 어떻게 처리할지를 정하는 것이다. 비개발자가 가장 많이 빠뜨리는 부분이 바로 예외처리이다.

| 기능명 | 예외 상황 | 처리 방식 |
|---|---|---|
| 클래스 등록 | 필수값 누락 | 저장하지 않고 누락 항목 표시 |
| AI 문구 생성 | OpenAI API 실패 | 오류 메시지 표시, 다시 생성 버튼 제공 |
| 승인 요청 | 승인자 미지정 | 승인자 설정 안내 |
| 예약 게시 | 예약 시간이 과거 | 현재 이후 시간 선택 안내 |
| Facebook 게시 | Token 만료 | Facebook 재연동 안내 |
| 게시 실패 재시도 | 최대 재시도 초과 | 수동 게시 안내 |

### 강사용 강조 문장

> 좋은 기능명세서는 성공했을 때만 쓰는 문서가 아니라 실패했을 때도 어떻게 할지 쓰는 문서입니다.

| 클래스 등록 | 가능 | 가능 | 불가 | 조회 |
| AI 문구 생성 | 가능 | 가능 | 가능 | 불가 |
| 승인 요청 | 가능 | 가능 | 불가 | 불가 |
| 승인 처리 | 가능 | 불가 | 가능 | 불가 |
| Facebook 연동 | 가능 | 불가 | 불가 | 불가 |

## 6.10 저장 데이터 작성

저장 데이터는 기능 실행 후 데이터베이스에 남겨야 하는 정보이다.

### 예시: Facebook 예약 게시

| 저장 데이터 | 설명 |
|---|---|
| content_id | 어떤 콘텐츠인지 |
| version_id | 어떤 버전인지 |
| facebook_page_id | 어느 Page에 게시하는지 |
| scheduled_at | 예약 시간 |
| status | pending, success, failed |
| retry_count | 재시도 횟수 |
| last_error_message | 마지막 오류 메시지 |
| created_by | 예약한 사용자 |
| created_at | 예약 생성 시간 |

### 강사용 설명

저장 데이터를 쓰지 않으면 나중에 개발자가 DB 설계를 할 수 없다. 비개발자는 정확한 DB 용어를 몰라도 “무엇을 기록해야 하는지”는 설명할 수 있어야 한다.

## 6.12 검수 기준 작성

검수 기준은 기능이 제대로 만들어졌는지 확인하는 기준이다.

### 작성 공식

> [조건]에서 [행동]을 했을 때 [기대 결과]가 나오면 정상으로 본다.

| 기능명 | 검수 기준 |
|---|---|
| AI 홍보문구 생성 | 클래스 정보를 선택하고 생성 버튼을 누르면 홍보문구 3안이 표시된다. |
| 승인 요청 | 승인 요청 후 콘텐츠 상태가 `review_requested`로 변경된다. |
| 예약 게시 | 승인 콘텐츠를 미래 시간으로 예약하면 `publishing_jobs`에 `pending`으로 저장된다. |
| 게시 실패 재시도 | 실패 상태 작업에서 재시도 버튼을 누르면 `retry_count`가 증가한다. |
| 권한 제한 | Viewer는 콘텐츠 생성 버튼을 사용할 수 없다. |
